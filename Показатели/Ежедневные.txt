Коворкинг

1. Что реально анализируем в коворкинге

Фиксируем принцип:
	•	Направления не сравниваются между собой как доли структуры.
	•	Каждое направление анализируется само по себе во времени.
	•	Единственные осмысленные связи:
	1.	Лаборатория ↔ Общий зал
	2.	Загрузка ↔ Выручка
	3.	Загрузка коворкинга ↔ Выручка кофейни (межблочная связь)

⸻

2. Что собираем ежедневно (сырые данные)

По коворкингу
	1.	revenue_open_space — аренда общего зала
	2.	revenue_cabinets — аренда кабинетов
	3.	revenue_lab — выручка лаборатории
	4.	revenue_retail — выручка ритейла

По загрузке
	5.	load_percent — процент загрузки площадки за день
(или средняя загрузка за день)

По кофейне (из соседнего блока)
	6.	coffee_revenue_total — выручка кофейни за день

⸻

3. Какие показатели считаем автоматически

Базовая динамика (по каждому направлению отдельно)
	1.	revenue_open_space
	2.	revenue_cabinets
	3.	revenue_lab
	4.	revenue_retail

Только:
	•	по дням
	•	по неделям
	•	по месяцам

Без долей, без микса.

⸻

Единственное относительное соотношение внутри коворкинга

Ключевой показатель
	5.	lab_to_open_space_ratio = revenue_lab / revenue_open_space

Это единственная доля, которую имеет смысл считать.

Смысл:
	•	падает → лаборатория монетизируется хуже, чем основной поток
	•	растёт → лаборатория усиливает LTV резидентов

⸻

Связь загрузки и денег
	6.	revenue_per_load = revenue_open_space / load_percent
(или просто корреляция load_percent ↔ revenue_open_space)

И базовые пары для анализа:
	•	load_percent ↔ revenue_open_space
	•	load_percent ↔ revenue_lab
	•	load_percent ↔ revenue_retail

⸻

Связь коворкинга и кофейни
	7.	coffee_revenue_total
	8.	coffee_revenue_per_load = coffee_revenue_total / load_percent

Ключевые пары:
	•	load_percent ↔ coffee_revenue_total
	•	revenue_open_space ↔ coffee_revenue_total

⸻

4. Какие агрегаты строим

Дневные
	•	все показатели по дням

Недельные (основной уровень)
	•	сумма за неделю:
	•	revenue_open_space
	•	revenue_cabinets
	•	revenue_lab
	•	revenue_retail
	•	coffee_revenue_total
	•	средние за неделю:
	•	load_percent
	•	lab_to_open_space_ratio

Месячные
	•	те же агрегаты за месяц

⸻

5. Какие сравнения используем

Только простые временные сравнения, без структуры.

Для каждого направления отдельно:

1) Текущий период vs предыдущий
	•	Эта неделя vs прошлая неделя
	•	Эти 4 недели vs предыдущие 4 недели

2) Текущий период vs прошлый год (YoY)
	•	Те же недели прошлого года

3) Текущий период vs лучший период
	•	Лучшие 4 недели за 12–24 месяца
(по каждому направлению отдельно)

⸻

6. Какие сигналы фиксируем (только управленчески осмысленные)

По каждому направлению отдельно
	1.	revenue_direction:

	•	|Δ| ≥ 12%
	•	и ≥ 2 периодов подряд

Сигналы:
	•	падает аренда общего зала
	•	падают кабинеты
	•	падает лаборатория
	•	падает ритейл

Без попытки объяснять через другие направления.

⸻

Ключевой сигнал по лаборатории
	2.	lab_to_open_space_ratio:

Порог:
	•	падение ≥ 15%
	•	и ≥ 2 периодов подряд

Смысл:
	•	лаборатория недомонетизирует текущий поток
→ проблема ассортимента, цен, наличия, сервиса

Это главный управленческий индикатор по лаборатории.

⸻

Ключевые сигналы по загрузке
	3.	Высокая загрузка + низкая выручка

Фиксация:
	•	load_percent ≥ среднее за 8 недель
	•	и revenue_open_space ≤ среднее за 8 недель − 10%

Смысл:
	•	много людей, мало денег
→ пакеты, абонементы, смещение в “некассовую” загрузку

Это не ошибка, а триггер “обратить внимание”.

⸻

Ключевые сигналы по связке с кофейней
	4.	Высокая загрузка + слабая кофейня

Фиксация:
	•	load_percent ≥ среднее за 8 недель
	•	и coffee_revenue_total ≤ среднее за 8 недель − 10%

Смысл:
	•	резиденты есть, но не покупают кофе
→ проблема предложения, скорости, витрины, апселла

Это очень сильный межблочный сигнал.

Кофейня

1. Что собираем ежедневно (сырые данные)

Продажи
	1.	checks — количество чеков
	2.	revenue_total — общая выручка

Категории выручки
	3.	revenue_coffee_hot — кофе / горячие напитки
	4.	revenue_drinks_cold — напитки из холодильника
	5.	revenue_desserts — десерты
	6.	revenue_food — еда / несладкое

Для еды (в целом)
	7.	sold_food_total — всего продано еды (desserts + food)
	8.	written_off_food_total — общее списание еды

⸻

2. Какие показатели считаем автоматически

Базовые
	1.	avg_check = revenue_total / checks

Структура выручки (доли)
	2.	share_coffee = revenue_coffee / revenue_total
	3.	share_drinks_cold = revenue_drinks_cold / revenue_total
	4.	share_desserts = revenue_desserts / revenue_total
	5.	share_food = revenue_food / revenue_total

Для списаний (только общее по еде)
	6.	writeoff_rate_food = written_off_food_total / (sold_food_total + written_off_food_total)

4. Какие периоды сравниваем (строго фиксированный набор)

Использовать только эти 4 типа сравнений.

1) Текущий период vs предыдущий такой же
	•	Эта неделя vs прошлая неделя
	•	Эти 4 недели vs предыдущие 4 недели

2) Текущий период vs тот же период прошлого года (YoY)
	•	Те же недели прошлого года

3) Текущий период vs лучший исторический период (бенчмарк)
	•	Лучшие 4 недели за последние 12–24 месяца
(по revenue_total)

4) Внутренний микс
	•	Доли категорий сейчас vs:
	•	прошлый период
	•	прошлый год
	•	лучший период

⸻

5. Какие показатели с чем сравниваем

Для revenue_total
	•	vs прошлая неделя
	•	vs прошлый год
	•	vs бенчмарк

Для checks
	•	vs прошлая неделя
	•	vs прошлый год

Для avg_check
	•	vs прошлая неделя
	•	vs прошлый год
	•	vs среднее за последние 8 недель

Для категорий (coffee / drinks_cold / desserts / food)

Считать два типа показателей:

Абсолют
	•	revenue_category (руб)

Сравнивать:
	•	vs прошлая неделя
	•	vs прошлый год
	•	vs бенчмарк

Относительное
	•	share_category (%)

Сравнивать:
	•	vs прошлый период
	•	vs прошлый год
	•	vs лучший период

Для списаний еды (только общее)

Показатель
	•	writeoff_rate_food

Сравнивать:
	•	vs среднее за последние 8 недель
	•	vs норматив

⸻

6. Пороги сигналов (финальная версия)

Критичные отклонения
	•	revenue_total: |Δ| ≥ 10%
	•	checks: |Δ| ≥ 8%
	•	avg_check: |Δ| ≥ 8%
	•	share_category: |Δ| ≥ 5 п.п.
	•	writeoff_rate_food: > 10%

Условия фиксации сигнала

Сигнал фиксируется, если:
	•	отклонение ≥ порога
	•	и длится ≥ 2 периодов подряд
